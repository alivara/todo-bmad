## Epic Summary

Epic 4 automated the project's already-committed quality tooling and test suites on GitHub Actions, so every PR and every push to `main` is verified without manual steps — a green pipeline as the precondition for merge. It was deliberately sequenced ahead of resuming Epic 2–3 work so the safety net exists before more mutation paths are built. Pure infrastructure/quality: no product, schema, or wire-contract change.

**Process metrics:** each story ran through the unattended dev-auto loop (plan → implement → adversarial expert review → real-CI verification). The pipeline caught **4 distinct real defects on its own PRs before merge** — see below.

## What Went Well

- **The real CI run was the authoritative check.** Repeatedly, a locally-green change failed on a clean CI runner — and that's the system working. It caught what pre-warmed local environments hid.
- **Adversarial expert review (Gopher + Pixel) caught gate-honesty bugs no automated check would.** The standout: the Go test-report step's trailing `|| true` clobbered `${PIPESTATUS[0]}`, so a stderr-only failure (e.g. a stale `go.sum`) would have exited 0 and merged green. A skeptical reviewer found it; no linter would.
- **Scope discipline held.** Zero product-code change across all three stories; the only deltas were the workflow, two pre-existing test-locator fixes (4.2), and README docs (4.3).
- **Worktree isolation enabled safe parallel work.** Three stories in flight without disturbing other worktrees' running stacks (one `WEB_PORT` remap avoided a port collision).
- **Fast convergence via honest failures.** Each red CI run produced a precise, actionable fix, so iterations were short.

## What Was Harder / Lessons Learned

- **Grounding assumptions must be verified against source, not assumed.**
  - `devices['iPhone SE']` defaults to **WebKit**, not mobile Chromium — the initial 4.2 investigation got this wrong, so CI installed only chromium and every `e2e-mobile` test failed on a clean runner. Cost one red round.
  - golangci-lint prebuilt binaries lag the Go toolchain: a binary built with Go 1.24/1.25 **refuses** to lint a `go 1.26` module. Fixed in 4.1 by pinning `golangci-lint-action@v7` + `install-mode: goinstall` (build from source with the runner's Go 1.26).
- **"Local green" is a hypothesis; the real pipeline is the proof.** The most reliable pattern all epic was: implement → verify locally → *and then trust the actual PR CI run as the gate*. Local success hid browser-cache and tool-version differences.
- **A self-inflicted fix caught before shipping:** the first 4.2 locator fix used `getByRole('alert', { name })`, but the `alert` role has name-from-content=false so it matched nothing. Local re-verification caught it before it merged — the good version of a mistake. Final fix: `getByRole('alert').filter({ hasText: /…/ })` (text-content match, role-preserving).
- **CI docs must be executed, not described.** In 4.3, reviewers found the "run the same checks locally" commands didn't actually reproduce CI (missing `npx playwright install`, a golangci-lint version caveat, the Go `testseed` integration command). Documentation of commands needs the same "does it actually run?" bar as code.

## Key Insights

1. The authoritative check is the real pipeline on a clean runner — it surfaces what warm local environments mask (browsers, tool versions, caches).
2. Adversarial review is where gate-honesty defects get caught (the `PIPESTATUS` clobber); automated checks won't find "the check silently passes when it shouldn't."
3. Verify environment/tooling facts against source (device browser type, tool build-Go version) — a single wrong assumption costs a full CI round.
4. Treat command documentation as executable: if a doc says "run X locally to reproduce CI," X must actually reproduce CI.
